# Complexity gates

Use these as candidate signals, never automatic findings. Admit an item only through the burden, current-justification, reachability, simpler-alternative, behavior-preservation, proportionality, and exception gate in `SKILL.md`.

| Surface | Candidate signals | Justified exceptions and boundary checks |
|---|---|---|
| **Source and local structure** | One-use pass-through helpers; trivial wrappers; type guards that merely rename an existing check; tiny files that add navigation; speculative factories, adapters, options, generics, hooks, or extension points; condition-laden shared abstractions; duplicate state or ownership. | Keep intention-revealing domain language, useful type narrowing, stable policy, real module or trust boundaries, externally imposed contracts, and structure that makes lifecycle or cleanup ownership explicit. Established duplication may justify a shared abstraction; incidental similarity may justify local logic instead. |
| **Plans and proposals** | Work for hypothetical users or scale; unrequested flexibility; premature migration, rollout, plugin, fallback, or generalization phases; safeguards without a reachable failure path; validation machinery larger than the decision it supports. | Keep choices required by current acceptance criteria, known consumers, approved migration/rollout needs, destructive or irreversible effects, measured constraints, and explicit user or authority decisions. |
| **Architecture and data flow** | Layers that only forward; parallel models or duplicate sources of truth; adapters around one stable implementation; broad registries/service locators; abstractions whose parameters and branches encode unrelated cases. | Keep stable domain boundaries, centralized policy, externally imposed interfaces, real isolation or ownership needs, structured lifecycle/cleanup, and established duplication whose shared behavior is unlikely to diverge. |
| **Review findings and proposed fixes** | Recommendations that broaden scope, add frameworks or layers, harden unreachable paths, generalize a local fix, or add checks/state without naming the failure path and owner. | Retain fixes for evidenced defects, required authority coverage, trust-boundary or data invariants, measured risks, and destructive/non-idempotent side effects. Preserve the supplied finding ID and let the owner decide; a reviewer recommendation creates no work automatically. |
| **Tests and validation machinery** | Broad test DSLs; deep fixtures; excessive mocking; combinatorial matrices without distinct behavior; duplicated assertions; helpers that hide setup or outcomes; tests for speculative configuration combinations. | Keep tests that uniquely protect behavior, wiring, parsers, compatibility, security/data invariants, concurrency, cleanup, upgrade canaries, or irreversible effects. Repetition can be clearer than a conditional test abstraction. |
| **Configuration** | Unused flags, modes, environment variables, nested overrides, or configuration combinations; invalid combinations; knobs with one fixed deployment value; configuration that duplicates source state. | Keep externally imposed deployment contracts, materially distinct supported environments, safe operational controls, secret boundaries, and settings with named owners and exercised combinations. |
| **Dependencies and technology** | Novelty dependencies; libraries used for trivial local behavior; overlapping packages; new infrastructure before the existing stack is shown insufficient; wrappers preserving no meaningful boundary. | Keep dependencies with demonstrated capability, security, compatibility, maintenance, or total-ownership advantage. Account for replacement cost, ecosystem conventions, licensing, updates, and operational support rather than preferring either dependency-free or novel technology by default. |
| **Concurrency and scale** | Unmeasured queues, caches, retries, locks, workers, batching, sharding, or backpressure; coordination for hypothetical traffic; state machines for execution that is actually serial. | Complexity is justified only by measured constraints or a realistic overlap model. A concurrency concern requires **realistic overlapping execution + shared mutable state or a non-idempotent side effect + meaningful harm**. User count alone proves neither safety nor danger. Keep structured concurrency, cancellation, cleanup, reconciliation, and idempotency where the execution model or side effects require them. |
| **Validation and security** | Repeated caller-and-callee checks; ceremonial validation after trusted parsing; scattered authorization policy; defensive branches with no reachable input or failure source. | Keep validation at real user, external, persistence, message, or file trust boundaries; preserve authentication, authorization, data invariants, safe parsing, and defense required by the threat model. Prefer carrying validated knowledge over deleting necessary controls. Security labels alone do not prove either necessity or excess. |
| **Fallbacks and compatibility** | Fallback-over-fallback chains; silent alternate behavior; unsupported legacy formats or clients; compatibility adapters without an identified consumer, contract, or removal condition. | Keep meaningful compatibility commitments, externally imposed contracts, staged migrations, recovery paths, and fallbacks tied to observed operational failures. Validate ambiguous consumer or rollback needs before acting. |
| **Operations and resilience** | Unmeasured retries, caches, queues, circuit breakers, feature flags, dashboards, logs, recovery jobs, or deployment layers; duplicated observability; safeguards without an operator action. | Keep controls for measured service limits, required availability, useful operator decisions, destructive/irreversible side effects, unknown outcomes, reconciliation, rollback, and meaningful recovery. Ownership and operational evidence matter more than component count. |

## Interpretation rules

- Do not apply blanket helper-count, line-count, DRY, defensive-code, security, dependency, or concurrency rules.
- Compare total conceptual, maintenance, migration, validation, and operational burden—not merely local code size.
- Prefer `delete → direct use → local logic/helper → existing shared abstraction → new abstraction`, but retain a later option when evidence shows it is the smallest safe boundary.
- If current need, reachability, authority, compatibility, or behavior preservation remains materially uncertain, recommend bounded validation or a user decision instead of asserting removal.
