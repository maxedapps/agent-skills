# Amazon Linux 2023 on EC2 profile

Load only after inspection classifies the host as `al2023-ec2`. Supported: `ID=amzn`, `VERSION_ID=2023`, positively identified EC2, systemd, standard `sshd_config.d` include, `x86_64`/`aarch64`.

## Fixed contract

| Area | Behavior |
|---|---|
| Updates | One concrete approved `releasever`; base phase upgrades and pins to it; no floating `latest`; no Ubuntu-equivalent unattended-update claim |
| Ingress | EC2 Security Groups only, every run—including without Docker |
| Docker | Amazon repository `docker` package only; Engine, Compose, Buildx, `hello-world` required; no Docker CE fallback |
| Node.js | Amazon `nodejs24`/`nodejs24-npm` or `nodejs22`/`nodejs22-npm`; namespaced binaries via `alternatives` |
| Tailscale | Official `amazon-linux/2023` stable repository; default `tailscale up --accept-dns=false --ssh=false` |

There is no AL host-firewall or public-SSH closure script. Closure is always a Security Group change confirmed by the user.

## Inspect release candidates

Inspection reports current and available AL2023 versions via `dnf check-release-update` when available. Before base setup:

1. Select one concrete target release (for example `2023.6.20241010`)—never `2023` or `latest`.
2. Read that release's notes from current AWS documentation.
3. Stop for explicit user approval of that exact `releasever`.

## Base setup

```bash
sudo ./scripts/al2023-ec2/apply-base-setup.sh --releasever <approved-concrete-version>
```

The script rejects empty values, `2023`, and `latest`. After upgrade it verifies:

```bash
rpm -q system-release --qf '%{VERSION}\n'
```

equals the approved release. Later installers use the host's pinned repository release automatically—they take no `--releasever` argument.

Do not configure `dnf-automatic`. Do not claim future releases apply automatically. Report later `dnf check-release-update` results as manual maintenance.

## Administrator and SSH

```bash
sudo ./scripts/configure-admin-user.sh --admin <admin-user>
# Gate: fresh ssh + sudo -n true
sudo ./scripts/harden-ssh.sh --admin <admin-user> --confirmed-login-tested
# Gate: fresh ssh + sudo -n true again
```

Prefer existing `ec2-user` when it already meets the intended-admin criteria.

## Tailscale (optional)

```bash
sudo ./scripts/al2023-ec2/install-tailscale.sh
```

### Gate: authenticate (DNS-safe default)

AL2023 has a documented Tailscale DNS forwarding-loop risk. Default:

```bash
sudo tailscale up --accept-dns=false --ssh=false
```

Changing resolver mode or enabling MagicDNS is separate requested work and must be deliberately established and tested. Verify status, addresses, `RunSSH: false`, and that host DNS resolution still works. Independent OpenSSH-over-Tailscale test before SG public SSH closure.

## Ingress — Security Groups only

1. Confirm the correct Security Group is attached to this instance (user confirmation; local output cannot prove attachment intent).
2. Consult current AWS Security Group documentation. Never request API tokens.
3. Ensure rules cover public IPv4 and IPv6 as assigned.
4. Allow only approved ports (optional ICMP/ICMPv6; optional UDP 41641 for direct Tailscale; approved app ports).
5. For tailnet-only SSH: after the Tailscale login gate, remove public TCP `<ssh-port>` for `0.0.0.0/0`, `::/0`, and other public sources.
6. Stop while the user applies and confirms the SG change.
7. Test fresh intended-path success and public IPv4/IPv6 failure with connection sharing disabled.

When public SSH remains intended, retain TCP `<ssh-port>` only from explicitly approved sources when practical and test a fresh public login after changes.

Explain accurately: sshd may still listen on public interfaces; the SG blocks reachability. Detaching or misapplying the SG can expose key-only SSH and public Docker bindings. Provider console/rescue is the recovery path.

## Docker (optional)

Only after SG ingress is established and externally tested:

```bash
sudo ./scripts/al2023-ec2/install-docker.sh
sudo docker info
sudo docker compose version
sudo docker buildx version
```

Amazon's package must satisfy Engine + Compose + Buildx + `hello-world`. If it lacks a required component, stop with no Docker CE/plugin fallback.

Optional root-equivalent group (separate approval):

```bash
sudo ./scripts/al2023-ec2/install-docker.sh --admin <admin-user>
```

## Node.js (optional)

```bash
sudo ./scripts/al2023-ec2/install-nodejs.sh          # major 24
sudo ./scripts/al2023-ec2/install-nodejs.sh --major 22
```

Verify both default and namespaced commands, and `alternatives --display node` consistency.

## Verification

Always external/SG mode. After independent intended-access and public IPv4/IPv6 tests:

```bash
sudo ./scripts/verify-setup.sh \
  --profile al2023-ec2 \
  --admin <admin-user> \
  --ssh-access <public|tailscale-only> \
  --ingress external \
  --confirmed-security-group-tested
```

Profile checks include required SG confirmation evidence flag, SELinux mode reported accurately (permissive is a warning, not a silent pass claim), selected/current release version, `smart-restart` / `needs-restarting -r` reboot markers, and no automatic-update claim.

## Maintenance note

AL2023 locks instances to versioned repositories. Future releases from `dnf check-release-update` are manual maintenance: read notes, approve a concrete version, upgrade deliberately. Keep the approved release version in the run record and final report.
